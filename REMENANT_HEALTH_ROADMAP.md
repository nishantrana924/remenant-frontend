# Remenant Health: Senior Development & Stability Roadmap

## 1. Executive Summary
The Remenant Health platform currently suffers from **Architectural Desync** and **Global Style Pollution**. The primary instability arises from the interaction between **Unpoly** (partial page updates) and **Vite** (versioned assets), leading to frequent 404 errors and layout breakage after updates. This document outlines a systematic, enterprise-grade sprint plan to transform the platform into a stable, scalable, and secure eCommerce system.

---

## 2. Root Cause Analysis (RCA)

### 2.1 Asset Synchronization Gap
- **Problem**: Vite generates hashed filenames for cache-busting. Unpoly swaps content fragments without refreshing the `<head>`. 
- **Root Cause**: The system lacks an automated mechanism to notify the frontend when a new build has invalidated the current assets.
- **Impact**: **CRITICAL**. CSS and JS 404 errors causing total UI failure after deployment.
- **Risk**: User abandonment due to broken interface.

### 2.2 Global CSS Regression
- **Problem**: Direct tag styling (`h1`, `h2`, `p`, `span`) in `app.css`.
- **Root Cause**: Lack of CSS encapsulation. Styles intended for specific sections bleed into global components.
- **Impact**: **HIGH**. Visual inconsistencies and "chain reaction" regressions (fixing one page breaks another).

### 2.3 Middleware & Auth Fragility
- **Problem**: Conflicting `AdminMiddleware` and `EnsureUserIsAdmin`.
- **Root Cause**: Overlapping logic and non-standard redirect routes (e.g., `user.dashboard`).
- **Impact**: **MEDIUM**. Potential unauthorized access or infinite redirect loops.

## 4. Implementation Strategy: Critical Stability

### 4.1 Automated Asset Versioning (TASK-001)
*   **Strategy**: Create a helper to read the `manifest.json` generated by Vite.
*   **Step 1**: In `HandleUnpolyRequests` middleware, calculate `md5_file(public_path('build/manifest.json'))`.
*   **Step 2**: Set the `X-Up-Assets-Revision` header to this hash.
*   **Step 3**: Unpoly will detect the change and trigger a clean full-page reload on the next navigation, ensuring the browser always has the correct CSS/JS hashes.

### 4.2 Security Hardening (TASK-002)
*   **Strategy**: Remove "Web Artisan" exposure.
*   **Action**: Delete `migrate`, `seed`, and `setup` routes from `web.php`. Use CLI only for these operations.
*   **Protection**: Ensure `.env` is blocked at the server level (Nginx/Apache) and not exposed via any public route.

## 5. Refactoring Plan

### 5.1 Frontend (UI/UX Consistency)
- **CSS Cleanup**: Move all base typography to Tailwind's `@layer base`. Remove `!important` flags where possible.
- **JS Consolidation**: Move inline scripts from `app.blade.php` into `app.js`. Use Unpoly's `up.compiler` or `up.on('up:fragment:inserted')` for re-initialization.
- **Icon Strategy**: Standardize on a single Lucide initialization point to prevent "flicker" during partial loads.

### 5.2 Backend (Performance & Logic)
- **Service Layer**: Implement `OrderService` and `PaymentService` to handle Razorpay complexity.
- **Model Scopes**: Use Eloquent scopes for common queries:
  ```php
  public function scopePublished($query) { return $query->where('status', 'published'); }
  ```
- **Review Optimization**: Replace looped queries in `ProductController` with a single aggregation query:
  ```php
  $breakdown = $product->reviews()->selectRaw('rating, count(*) as count')->groupBy('rating')->pluck('count', 'rating');
  ```

## 8. Deployment Strategy
1. **Pre-Deploy**: Run `npm run build` and verify manifest integrity.
2. **Deploy**:
   - `php artisan down`
   - `git pull`
   - `composer install --optimize-autoloader`
   - `php artisan migrate --force`
   - `php artisan up`
3. **Post-Deploy**: Clear Unpoly cache and verify asset hashes in the browser console.
